Wi-Fi 1-day 취약점 PoC

CVE-2022-47522

Wi-Fi 네트워크에서 Station이 네트워크 외부의 서버로 요청을 보내는 경우, 요청과 응답 사이에 지연 시간이 발생하는데, 이 타이밍에 공격자는 AP와 Station의 연결을 끊고 응답을 가로챌 수 있다.

원인

AP가 Frame을 routing할 때 인증 정보를 활용하지 않고 MAC주소만을 이용하기 때문에 발생하는 취약점이다. Station이 AP에 연결할 때 적절한 사용자 인증(ID와 Password, 인증서 등)을 거친다 해도, 공격자가 이 연결을 끊고 해당 Station(Victim)의 MAC주소를 사용하여 AP에 연결하면 Victim과 인증 정보가 일치하지 않아도 Frame을 받을 수 있다.

조건

이 취약점은 프로토콜의 설계 결함에 의한 것이므로, WPA 버전, 인증 방식 또는 특정 구현체와 관계없이 이론상 모든 Wi-Fi 네트워크에서 발생한다. 특히, 클라이언트 사이의 공격을 방지하는 Client Isolation이 적용된 경우에도 취약하다.

메모리 취약점 탐지 도구 개발

주제

LLVM Pass를 활용하여, C/C++로 작성된 프로그램의 메모리 취약점을 런타임에 탐지하는 도구 개발

목표

런타임에 다음의 메모리 버그를 탐지할 수 있도록 컴파일 타임에 코드 자동 변환

  • Out-of-Bounds Access
  • Use-After-Free

사전지식(LLVM이란?)

LLVM은 다양한 소스 언어와 아키텍처에 대해 통합된 중간 표현(IR, Intermediate Representation)을 사용하는 컴파일러 프레임워크로, 컴파일러 개발의 복잡도를 낮추어 준다. 이 구조 덕분에 언어 및 플랫폼에 독립적인 분석과 최적화가 가능하다.

LLVM Pass는 이러한 LLVM IR을 대상으로 코드 변환이나 최적화를 수행하는 모듈로, 정적 분석, 코드 계측, 취약점 탐지 등의 작업에도 활용된다. 본 프로젝트에서는 LLVM Pass를 활용해 프로그램의 IR을 분석하고, 메모리에 접근하는 코드를 추적하여 잠재적인 취약점을 탐지하고자 했다.

주요 내용 및 핵심 기능

Out-of-Bounds Access 탐지

Softbound 라이브러리 개발

  • 프로그램 내의 모든 포인터에 대해 해당 메모리 블록의 경계 정보를 나타내는 메타데이터를 관리
  • 메모리 접근 시 메타데이터를 참조하여 경계를 넘어서는지 검증
  • 기존 도구인 AddressSanitizer가 탐지하지 못하는 케이스 탐지 가능
    • 구조체 내부의 필드 사이에서 발생하는 OOB 탐지
    • AddressSanitizer의 Redzone을 넘어서 발생하는 OOB 탐지
Use-After-Free 탐지

Shadow memory 라이브러리 개발

  • 프로그램 메모리 주소와 8:1로 매핑되는 Shadow memory 할당
    • Shadow memory의 값은 이에 매핑된 실제 메모리 주소의 상태(할당, 해제 등)를 나타냄
  • 메모리 할당, 해제 시 Shadow memory에 해당 주소의 상태 업데이트
  • 메모리 접근 시 Shadow memory를 참조하여 접근 가능한 영역인지 검증
코드 변환 및 삽입

LLVM Pass

  • 타겟 프로그램의 컴파일 과정에서 소스코드(정확히는 LLVM IR)에 대해 LLVM Pass 실행
  • 메모리 할당, 해제, 접근, 포인터 연산 등을 식별하여 적절히 변환하고, 위 두 라이브러리의 함수를 적절히 삽입

변환된 타겟 프로그램을 실행하여 메모리 버그를 탐지하면 에러를 발생시킴

역할

  • Shadow memory 라이브러리 개발
  • Use-After-Free를 탐지하는 LLVM Pass 작성

TARA(Threat Analysis and Risk assessment)

목적

자동차의 OTA(Over-the-Air) 업데이트 시스템을 대상으로 위협 모델링과 위험도 평가를 수행하고, 잠재적인 보안 위협에 대한 체계적인 분석을 통해 안전한 업데이트 환경을 위한 보안 요구사항 도출

주요 내용

  • 아이템 정의: 분석 대상인 OTA 업데이트 시스템의 범위 및 구성 요소 정의
    • 아이템 경계 설정, 운영환경 가정
  • 자산 식별: OTA 업데이트 시스템 내 보호가 필요한 정보 및 구성 요소 식별
    • MECE(Mutually Exclusive, Collectively Exhaustive) 원칙에 맞게 분류
  • 피해 시나리오 작성: 각 자산별로 발생 가능한 피해 시나리오 작성 및 영향 등급 산정
    • 영향등급은 안전, 재무, 작동, 개인정보 각각에 대해 산정 후 통합
  • 위협 시나리오 작성: 공격자가 취할 수 있는 위협 시나리오 도출
    • 각 시나리오를 STRIDE 모델 기반으로 작성
    • 침해 되는 보안 요소(기밀성, 무결성, 가용성) 명시
  • 공격 경로 분석: 위협에 대한 구체적인 공격 경로 분석
  • 공격 실현성 평가: 각 공격 시나리오에 대해 소요 시간, 필요한 전문성, 지식(정보), 기회(접근성), 필요 장비 등을 기반으로 실현 가능성 평가
  • 리스크 평가: 피해 영향 등급과 공격 실현성을 바탕으로 리스크 값 산정
  • 리스크 처리 방안: 도출된 리스크에 대한 대응 및 보안 요구사항 제안
    • 리스크 값이 낮아 위험을 보유, 공유 혹은 전가하기로 결정한 경우, 사이버 보안 Claim 작성
    • 리스크 값이 높아 위험을 감소하기로 결정한 경우, 사이버 보안 목표와 Control 조치 작성
